# 通用开发规则与最佳实践（AGENTS）

## 1. 通用开发规则

- 需求先行：实现前明确目标、范围、约束、验收标准。
- 单一事实源：状态、配置、规则必须有唯一可信来源。
- 职责分离：业务逻辑、状态管理、展示层、持久化清晰分层。
- 小步迭代：每次改动聚焦一个主题，降低回归风险。
- 可回归优先：每次改动都应可验证、可回滚、可追踪。

## 2. 工程最佳实践

- 逻辑尽量纯函数化，减少隐式副作用。
- 外部依赖（时间、随机、存储、网络）优先封装，便于测试。
- 公共接口变更时，同步更新调用方和文档。
- 功能变更必须同步更新测试，覆盖边界和失败路径。
- 文档必须解释“做了什么”和“为什么这么做”。

## 3. Git 协作规范

- 使用分支开发，避免高风险改动直接落主线。
- 提交信息清晰，建议 `type: summary`。
- 提交前检查 diff 范围，确保只包含本次改动。
- 合并前保证构建与测试通过。
- 冲突处理先统一规则再合并，避免语义倒退。

## 4. 过程注意事项

- 不并行执行会争抢锁的版本控制写操作（如 add/commit/merge）。
- 统一编码为 UTF-8，避免跨平台乱码。
- 构建产物、依赖目录、缓存文件不要纳入版本管理。
- 对 Shell、路径、权限差异保持敏感，必要时分步执行。
- 大改优先最小可行版本，再增量增强。

## 5. 开发环境发布经验（Dev）

- 开发环境目标是“快反馈”，不是“高稳定”：优先保留热更新和调试能力。
- 开发发布入口固定：统一脚本（如 `npm run dev`）和固定端口约定。
- 本地容器化用于环境一致性，不替代调试便利性。
- 对外依赖不稳定时，准备镜像源/代理备选方案，避免开发阻塞。
- 开发环境变更无需复杂审批，但必须可快速回退。

## 6. 测试环境发布经验（Test/Staging）

- 测试环境目标是“接近生产”：构建方式、运行方式、配置注入尽量与生产一致。
- 发布必须可复现：固定镜像标签、固定构建步骤、记录发布版本。
- 部署后先做基础健康检查：容器状态、端口、HTTP 状态码、关键路径可用性。
- 测试环境应支持快速重建：一条命令可拉起（如 compose/up）并可一键清理。
- 网络与权限是常见阻塞点：提前验证镜像仓库连通性、认证状态、端口占用。

## 7. 发布前检查清单

1. 需求与实现一致，验收标准可执行。
2. 构建通过，关键测试通过。
3. 文档更新完成（功能、限制、部署方式）。
4. 变更范围清晰，提交记录可追溯。
5. 当前分支工作区干净，可安全合并或发布。

## 8. 默认发布策略（Web）

- 对于 Web 项目，默认发布策略为：
- 正式环境：`github.io`（GitHub Pages）
- 开发环境： github.dev（本地开发环境/开发分支环境）
- 除非用户明确提出额外发布要求（例如自定义域名、云服务、灰度、多环境矩阵），否则按上述默认策略执行。
